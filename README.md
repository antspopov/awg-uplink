# AWG Split Gate

[English](#english-summary) · **Русский**

*Человекочитаемое имя продукта. Репозиторий на GitHub, каталоги установки, имена systemd-юнитов и сетевой интерфейс туннеля по-прежнему используют технический префикс **`awg-uplink`** — это не ошибка.*

---

## Краткое описание

**AWG Split Gate** — это набор сценариев и **веб-интерфейс управления** для сервера на Linux, который объединяет в одном месте: **раздельный egress/ingress**, **туннель AmneziaWG** (интерфейс `awg-uplink`), **политику маршрутизации** (включая georouting по IP и по доменам), **DNS** (dnsmasq, dnscrypt-proxy, интеграция с AmneziaDNS), **nftables-брандмауэр**, а также **MTProto-прокси** (установка и обслуживание через Web UI и `mtbuddy`).

Проект рассчитан на **чистую установку под root** через скрипт **`awg-webui-bootstrap.sh`**: он копирует файлы в `/opt/awg-uplink/webui`, поднимает **nginx** с HTTPS (Let's Encrypt или самоподписанный сертификат), настраивает **systemd**-юниты и создаёт конфигурацию в **`/etc/awg-uplink-webui/`**. Дальнейшая настройка выполняется **из браузера**: мониторинг, интерфейсы, туннель, роутинг, DNS, MTProto — без ручного редактирования десятка разрозненных конфигов (хотя продвинутые правки по-прежнему возможны).

## Требования

- **ОС:** в первую очередь **Ubuntu / Debian** (скрипт ставит пакеты через `apt-get`, при необходимости — **AmneziaWG** из PPA или сборкой из исходников).
- **Права:** установка только **от root** (`sudo ./awg-webui-bootstrap.sh`).
- **Сеть:** для профиля с **Let's Encrypt** домен должен указывать на сервер, порты **80/443** доступны извне; для **самоподписанного** сертификата браузер покажет предупреждение безопасности.
- **Репозиторий:** клон с **GitHub** — см. следующий раздел.

## Первый запуск: загрузка с GitHub и установка

### 1. Клонирование репозитория

На сервере (пример для ветки по умолчанию `main`):

```bash
sudo apt-get update -y
sudo apt-get install -y git

sudo git clone https://github.com/antspopov/awg-uplink.git
cd awg-uplink
```

Если нужна **конкретная ветка**:

```bash
sudo git clone --branch <имя-ветки> --single-branch https://github.com/antspopov/awg-uplink.git awg-uplink
cd awg-uplink
```

### 2. Запуск установщика

```bash
sudo ./awg-webui-bootstrap.sh
```

Скрипт:

- проверит наличие каталогов `webui/`, `lib/`, `systemd/`;
- при необходимости установит **AmneziaWG** (`awg-quick`, `awg`);
- скопирует веб-приложение в **`/opt/awg-uplink/webui`**, скрипты — в **`/usr/local/sbin/`**, unit-файлы — в **`/etc/systemd/system/`**;
- создаст **`/etc/awg-uplink-webui/`** с начальными `webui.env`, `georouting.json`, `dns.json` (если их ещё нет);
- настроит **DNS** (в т.ч. отключение stub listener **systemd-resolved** в пользу локального резолвера, см. комментарии в скрипте);
- включит и запустит связанные **systemd**-сервисы и таймеры;
- проведёт **интерактивный мастер**: профиль HTTPS (Web UI + MaskingTLS MTProto), домен, логин/пароль Web UI, при необходимости — **Let's Encrypt**;
- настроит **nginx** как reverse proxy к локальному Python-серверу и перезапустит nginx.

Опции скрипта (кратко):

| Опция | Назначение |
|--------|------------|
| `--no-start` | установить файлы и конфиги, но не перезапускать веб-сервис в конце |
| `--update-files-only` | только обновить файлы из репозитория и перезапустить `awg-uplink-webui`, без повторного мастера |

Справка: `sudo ./awg-webui-bootstrap.sh --help`

## После установки

1. **URL входа** — в конце мастера выводится сводка: режим HTTPS, **логин/пароль**, список URL по интерфейсам (часто `https://<ip>:<mask_port>` при самоподписанном сертификате). При **Let's Encrypt** основной адрес вида `https://<ваш-домен>:<mask_port>`.
2. Откройте ссылку в браузере → страница **входа** → дайджест-аутентификация (challenge-response), далее **cookie-сессия**.
3. Сервис веб-интерфейса: **`awg-uplink-webui.service`** (слушает `127.0.0.1` и порт из `webui.env`, наружу — **nginx**).
4. Первый шаг в UI: заполнить и **сохранить** блок **«Настройка интерфейсов»** — до этого остальные разделы (туннель, роутинг, DNS, MTProto и т.д.) **заблокированы** подсказкой-оверлеем.

## Обновление файлов без полного мастера

После `git pull` в каталоге репозитория на сервере:

```bash
cd /path/to/awg-uplink
sudo git pull
sudo ./awg-webui-bootstrap.sh --update-files-only
```

Так обновляются файлы приложения и перезапускается только **awg-uplink-webui**, без повторной настройки nginx/сертификатов через мастер.

## Важные пути и переменные

| Путь / переменная | Назначение |
|-------------------|------------|
| `/opt/awg-uplink/webui/` | Статика и `server.py` |
| `/etc/awg-uplink-webui/webui.env` | Параметры сервера UI: хост, порт, `AWG_UI_USER` / `AWG_UI_PASS`, `AWG_WEBUI_BASE_PATH`, домен и режим TLS для MTProto/Web, `AWG_UI_MASK_PORT` и др. |
| `/etc/awg-uplink-webui/interfaces.json` | Сохранённые egress/ingress, режим маршрутизации |
| `/etc/awg-uplink-webui/georouting.json` | Режимы georouting, списки, целевой интерфейс |
| `/etc/awg-uplink-webui/dns.json` | DNS upstream, dnscrypt, брандмауэр по портам, опции AmneziaDNS |
| `/var/lib/awg-uplink-webui/sessions.json` | Сохранённые сессии входа (чтобы после перезапуска UI не выкидывало из аккаунта до истечения срока) |
| `AWG_WEBUI_SESSIONS_FILE` | Переопределение пути к файлу сессий |
| `/etc/amnezia/amneziawg/awg-uplink.conf` | Конфиг туннеля (импорт из UI) |

## Функционал веб-интерфейса (для пользователя)

Ниже — разделы панели **в том порядке, в каком их удобно проходить при первой настройке**.

### Мониторинг сервера

- **CPU, RAM, сеть (RX/TX), load average, uptime** — обновление в реальном времени (короткий интервал опроса).
- Нужен для контроля нагрузки до и после включения туннеля, georouting и MTProto.

### Настройка интерфейсов

- **Egress** — основной выход в интернет: интерфейс, IPv4-адрес, шлюз, разрешённые **входящие TCP** на этом интерфейсе (например SSH).
- **Ingress** — отдельный вход, если есть второй публичный IP/интерфейс; иначе роль ingress может совпасть с egress (подсказки в форме).
- Кнопка **«Сохранить»** — записывает конфигурацию и запускает применение маршрутизации на стороне сервера.
- **Edit netplan** — просмотр/редактирование netplan (для опытных пользователей).

Пока egress (и при необходимости логика гейта) не удовлетворяют проверке готовности, **остальная панель** остаётся недоступной — это сделано специально, чтобы не строить маршрутизацию и VPN «в пустоту».

### Туннельный интерфейс (AmneziaWG / `awg-uplink`)

Доступен **после** сохранения настроек интерфейсов.

- **Импорт конфига** — выбор файла `.conf` / `.txt` или вставка текста в модальном окне, затем **«Импортировать»**.
- **«Перезапустить awg-uplink»** — перезапуск systemd-юнита туннеля и повторное применение связанной маршрутизации.
- Статус **UP/DOWN** отображается в шапке карточки; от него зависит доступность режимов **«маршрут в туннель»** и части проверок georouting.

### Роутинг

- **Маршрут по умолчанию:**  
  - **В Egress** — обычный выход через основной uplink.  
  - **В туннельный интерфейс** — весь исходящий трафик по умолчанию через **awg-uplink** (если интерфейс поднят).  
  - **Georouting** — выборочная маршрутизация по спискам (см. ниже).
- Переключение режима инициирует операцию на сервере; статус «применено / не применено» виден в интерфейсе.

### Настройка georouting

- Режимы **по IP** и/или **по доменам** (можно комбинировать в рамках выбранного сценария маршрутизации).
- **Цель маршрута для «списковых» ресурсов:** туннель или egress (взаимно согласовано с базовым маршрутом по умолчанию — подсказки в UI).
- **Готовые ссылки** на публичные списки (часть помечена как защищённая от удаления) + **свои include/exclude** построчно.
- Кнопка **«Применить»** — сохраняет `georouting.json`, обновляет списки и связанные **nft/ip rule** на сервере (через фоновые задачи, при необходимости с оверлеем ожидания).
- Для **доменного** режима важен работающий **DNS** и при доменной схеме — контейнер **AmneziaDNS** (см. DNS-панель и баннер про AmneziaVPN при отсутствии типичного Docker-стека).

### Настройка DNS

- **Состояние DNS** — агрегированный health: dnsmasq, dnscrypt-proxy, блоклисты, **Amnezia DNS**, блокировка DNS-транспорта и др.
- **Dnsmasq и dnscrypt-proxy** — выбор upstream, серверов dnscrypt, редактирование доменных списков (включая отдельное модальное окно для больших списков), сохранение с перезапуском служб.
- Опции **подмены AmneziaDNS** (Unbound → dnsmasq хоста) и **блокировки DNS-транспорта** (DoT/DoH и перенаправление UDP/TCP 53 в Docker-сети Amnezia) — с пояснениями в интерфейсе; требуют осознанного включения.
- Связь с **геомаршрутизацией по доменам**: при включённом доменном режиме без AmneziaDNS панель покажет предупреждения.

### Настройка MTProto

- **Установка / обновление MTProto** — асинхронная операция через API установки; отображается прогресс/фаза.
- **Пользователи** — добавление, редактирование, **включение/выключение**, удаление; копирование ссылок **tg://** и **t.me**; секреты отображаются в сокращённом виде.
- Защита **последнего активного пользователя** — нельзя выключить или удалить, если он единственный с секретом в `access_users` (и сервер, и UI это учитывают).
- **Исходящий интерфейс** MTProto (`direct` / egress / tunnel) — синхронизируется с `config.toml` и перезапуском сервиса прокси.
- **Правка `config.toml`** в модальном окне: подсказка, какие ключи может перезаписывать синхронизация Web UI; при открытом окне **опрос не перезатирает** текст (чтобы правки не «мигали»).
- **Masking health** — состояние nginx, таймера, endpoint маскировки и т.д. (зависит от профиля установки).

### Прочее в шапке

- **Баннер AmneziaVPN** — напоминание установить серверные компоненты через мобильное приложение, если не обнаружен ожидаемый Docker/bridge-стек.
- **«Выйти»** — завершение сессии Web UI.

## English summary

**AWG Split Gate** bundles routing, AmneziaWG tunneling (interface `awg-uplink`), optional georouting (IP/domain lists via nftables), DNS (dnsmasq + dnscrypt-proxy + AmneziaDNS hooks), firewall automation, and **MTProto** lifecycle into one **browser-based** control plane. Install from GitHub with **`sudo ./awg-webui-bootstrap.sh`** (HTTPS wizard, nginx, systemd). The UI is served behind nginx; start configuration from **Interfaces**, then tunnel, routing, DNS, and MTProto. Session cookies can be persisted to disk so a **webui service restart** does not always log you out. See the Russian sections above for paths, flags (`--update-files-only`), and detailed panel behavior.
